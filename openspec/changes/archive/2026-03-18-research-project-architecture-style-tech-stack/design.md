## Context

当前仓库使用 Next.js App Router 作为前端与页面层，使用 Hono 承载 `/api` 路由，数据访问由 Prisma + PostgreSQL 提供。代码已形成按职责分区的目录结构（`src/app`、`src/server`、`src/libs`、`src/database`），但尚未形成统一的“可复用研读与归档方法”，导致不同成员对模块边界和风格理解存在差异。

该变更目标不是改动运行时代码，而是产出一套标准化的基线研读与归档结果，并让该结果能够在后续需求分析与改造中被引用。

## Goals / Non-Goals

**Goals:**
- 建立统一研读维度：代码风格、封装逻辑、整体架构、技术栈。
- 定义固定证据来源：关键配置文件、核心目录、代表性服务与 API 路由实现。
- 输出可追踪归档：结论、证据、风险、建议与维护节奏。
- 让产物可直接支撑后续实现阶段（`tasks`）的拆解与验收。

**Non-Goals:**
- 不对业务功能、API 行为或数据库 schema 做直接修改。
- 不引入新的运行时依赖或重构现有目录结构。
- 不在本变更内制定完整代码规范细则（仅建立基线和差异观察）。

## Decisions

1. 采用“分层扫描 + 代表性样本”研读策略  
   - 决策：按 `app/server/libs/database/config` 分层扫描，并抽取代表文件形成证据。  
   - 原因：兼顾覆盖率与效率，避免只看单点文件导致结论失真。  
   - 备选：全量文件逐一阅读。  
   - 取舍：全量最完整但成本高，不适合首次基线建设。

2. 使用 capability 拆分归档范围  
   - 决策：将本次需求拆分为三个能力：架构基线、风格与封装审计、技术栈盘点。  
   - 原因：每个能力对应独立规范和后续任务，减少文档耦合。  
   - 备选：单一大而全文档。  
   - 取舍：单文档易写但后期维护和增量更新困难。

3. 归档输出以“事实 + 可执行建议”为核心  
   - 决策：每个能力要求同时记录观察事实、风险点与可执行后续动作。  
   - 原因：仅有描述无法直接驱动改进；建议可直接转为后续 change。  
   - 备选：仅记录现状。  
   - 取舍：仅现状缺少行动价值，难以形成持续优化闭环。

4. 约束验收标准为“可复核”而非“主观完备”  
   - 决策：要求结论均可关联到代码或配置证据，保证他人可复查。  
   - 原因：降低个人理解偏差，提升归档可信度。  
   - 备选：依赖经验总结。  
   - 取舍：经验总结快，但团队可复用性弱。

## Risks / Trade-offs

- [风险] 样本选择偏差导致结论不全面  
  → 缓解：明确最小覆盖面（关键目录 + 核心入口 + 代表模块）并保留“未覆盖范围”说明。

- [风险] 归档后快速过时  
  → 缓解：在任务中加入定期回顾与触发条件（如核心依赖升级、目录重构）。

- [风险] 建议过于抽象，难以执行  
  → 缓解：每条建议绑定优先级、影响范围与可落地动作。

- [权衡] 不改代码可降低实施风险，但短期无法消除现有技术债  
  → 缓解：通过后续 change 将高优先级问题转化为实施任务。
